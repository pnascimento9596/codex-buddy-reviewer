# Security and Privacy

## Trust boundaries

Untrusted inputs include source code, diffs, filenames, hook JSON, CLI values, Git output, provider stdout/stderr, worker summaries, and model-generated findings. Authentication state and repository contents are sensitive local inputs. The remote model provider is outside the local trust boundary.

## Current RC security status

A sealed whole-repository Deep Security Scan of the pre-fix `v0.5.0-rc.1` head found 20 findings: 17 medium and 3 low. The validated clusters covered credential syntax gaps, denied-content fragment and live Git metadata coverage, and lossy invalid UTF-8 Git pathname handling. Provider-free reproductions proved affected content in prepared external-review prompts or falsely complete evidence decisions.

The current working tree implements structural remediations and focused regression coverage for those families. The sections below document the intended security contract and the controls that currently exist. They do not claim final closure. Promotion to stable remains blocked until a fresh exact-final-tree sealed scan, complete cross-platform CI, and the host acceptance gates pass. A public RC source snapshot remains explicitly pre-final and does not claim those gates have closed.

## Consent and activation

- Both skills set `allow_implicit_invocation: false`.
- Manual review requires an explicit request.
- Automatic review is disabled by default and is enabled per canonical Git workspace.
- Enabling records a consent timestamp, the ordered primary and optional secondary reviewer descriptors, policy version, and monotonically increasing configuration revision.
- Worker-summary advisory egress is disabled by default and requires a second purpose-specific confirmation, primary provider/model binding, timestamp, policy version, and independent configuration revision. Immediately before egress, its exact consent record and bounded packet are revalidated and bound only to the primary reviewer's durable single-use capability under a short consent lock. The secondary reviewer always remains technical-only.
- Disabled mode captures no new baseline and makes no provider call.
- Hook installation requires Codex's separate hook trust review; changing `hooks/hooks.json` invalidates that trust hash.

## Evidence boundary

- Snapshot capture uses a private index/object directory and does not write user refs or the user index.
- `hash-object --no-filters` prevents repository-configured clean filters from executing.
- Capture is doubled and aborts on a changed tree, status, content fingerprint, or `HEAD`.
- Common credential paths and unsafe filenames are denied.
- Denied-source rename destinations, exact copies that match denied Git tree/index/worktree endpoints, and symlinks aimed at denied/absolute/escaping targets are conservatively denied. Both baseline and final endpoints are checked before a patch can be transmitted.
- Enumerated high-risk ignored-secret pathspecs are byte-fingerprinted locally so their endpoint exact copies are excluded. Automatic turn snapshots also maintain a private aggregate fingerprint for ordinary reviewable ignored files; a baseline-to-final change or unbounded inventory forces abstention without revealing the path. Manual one-shot review has no ordinary-ignored baseline signal and intentionally omits those files.
- Denied endpoints also contribute bounded, per-turn salted HMAC digests of normalized fragments. Long inputs use content-defined fragments. Denied inputs up to 32 normalized bytes use an exact length-bound fingerprint; inputs from 33 to 127 bytes contribute sliding 32-byte fingerprints searched across bounded candidate windows. Sliding-window entries retain a 128-bit HMAC prefix to keep the local inventory bounded. A digest collision can only exclude additional candidate content, not hide denied content. This catches exact small denied values and substantial values copied out of short records after NFC, whitespace, or CRLF changes without retaining or transmitting denied plaintext. Extracts shorter than 32 bytes from a longer short record are deliberately not indexed to avoid making ordinary code unreviewable through tiny-substring matches. Inventory, input, and short-search work caps fail closed when completeness cannot be established.
- Allowed-looking text is scanned locally for bounded high-confidence private-key/token formats and high-entropy values assigned to secret-like names. The scan is capped at 2 MiB; oversized or invalid UTF-8 candidates are excluded because scanning is incomplete. It is a targeted exfiltration guard, not universal secret classification.
- Direct content from denied paths is locally inventoried through one bounded privacy kernel and excluded from transmission when the current controls recognize it. The kernel covers exact values, normalized fragments, short excerpts, and registered live Git metadata sources under explicit source, byte, representation, and work limits. Any missing, unstable, incompatible, or over-budget source makes privacy coverage incomplete and blocks provider approval. An allowed-looking candidate can exist transiently in a mode-`0700` private object directory while local checks run; excluded candidates are removed during Stop cleanup.
- Patch and individual snapshot file sizes are bounded. One aggregate monotonic capture budget spans both stability passes and bounds elapsed time, paths, file bytes, Git input/output, Git operations, and private object bytes. Raw patches for incomplete/non-file evidence are not transmitted, and incomplete evidence prevents clean assurance.
- If the index and worktree hold different representations of one path, including a staged deletion restored as untracked content, Buddy marks that path `index_worktree_diverged`, sends neither representation, and abstains. Opaque dirty submodules likewise remain incomplete instead of collapsing to a false-clean tree.
- The original prompt is never sent to the independent reviewer. The worker summary is not sent unless its separate exact-purpose consent is enabled. Even with consent, Buddy evaluates the exact terminal-sanitized, at-most-4,000-character packet against the high-confidence secret scanner, exact excluded-path references, and the high-risk path policy immediately before capability issuance. An unsafe or incomplete assessment downgrades the review to technical-only without transmitting the summary. This is a conservative additional boundary, not semantic DLP.

## Provider isolation

- Windows v0.5 RC has an explicit platform privacy gate. Manual live review, automatic `UserPromptSubmit` and `Stop`, doctor provider checks, live evaluation, and the canonical provider dispatcher all fail closed before a turn snapshot, review prompt, capability, provider subprocess, or provider temporary run is created. Read-only diagnostics, pets, configuration, local dry runs, and offline validators remain available. This gate has no environment or CLI override; host-independent tests use an explicit in-process platform injection.
- Subprocesses use argv arrays with `shell: false`.
- Grok runs from synthetic `HOME` and `GROK_HOME` directories with only an explicit readable auth-file bridge. It runs one plan-only turn with memory, subagents, web, repository reads, shell, and writes disabled. On supported POSIX hosts, the required `--prompt-file` argument names `/dev/fd/0`. A fixed `/bin/sh`, `/bin/cat`, `/usr/bin/mkfifo`, and `/bin/rm` bridge converts supervised stdin into a private FIFO without interpolating provider arguments or creating a plaintext prompt file. The bridge waits for the producer and consumer independently, so either failure is authoritative. Missing bridge prerequisites fail closed during preflight. The adapter retains host-independent Windows transport code, but the platform privacy gate prevents live use of its temporary prompt path. Provider adapters create marked run directories under a dedicated parent and remove them after the call. A v2 marker binds the provider to a short display key and a full SHA-256 workspace binding without storing the raw path. POSIX identity-bound operations require a positive nanosecond creation time in addition to stable UID, mode, device, inode, marker, age, and dead-PID checks. An unsupported temporary filesystem fails closed before provider state is created. Windows stale deletion is disabled because the parent DACL is not yet verified.
- Grok preflight requires closed inventories and rejects active project instructions, hooks, plugins, MCP servers, skills, agents, or language servers. A missing binary, auth file, inventory field, transport envelope, or provable isolation boundary fails closed.
- Claude runs the authenticated Claude Code CLI in a disposable working directory with safe mode, a static reviewer-only system prompt that replaces dynamic per-machine default sections, strict empty MCP configuration, no tools, plan permission, no session persistence, an explicit JSON schema, and a bounded environment allowlist. This is the supported direct Claude Max route. Buddy transiently forwards only the allowlisted authentication variables required by the selected CLI process and does not deliberately persist or log their values. Claude Code authentication and administrator-managed policy remain outside Buddy's controllable customization boundary.
- Ollama history is disabled and only the prompt is sent through stdin. The CLI runs from an ownership-marked disposable directory rather than the reviewed repository, while retaining the existing authenticated home environment required by local and cloud connections. Local Ollama models receive the exact response schema through structured-output transport. Ollama Cloud models use JSON transport and the same strict local validation because the Cloud CLI path does not accept the full response schema.
- OpenCode runs a random-named agent in a disposable work, config, data, cache, state, and temporary tree. Project configuration, external skills, default plugins, Claude imports, MCP, sharing, auto-update, tools, and permissions are disabled. The selected `provider/model` prefix determines the one OpenCode auth entry projected into this environment. The rest of the OpenCode auth store and ambient provider credentials are not forwarded. Unknown events, tool-use events, error events, nonterminal text, and invalid JSON fail closed.
- ChatGPT Pro is supported through the OpenCode OpenAI OAuth connection. Kimi is supported only when its model is exposed through an already configured OpenCode provider connection. Direct Codex CLI and direct Kimi CLI execution remain unsupported until strict no-tools and no-inherited-context operation is proven while preserving the intended subscription authentication.
- Reviewer effort is validated against the selected adapter before mode persistence, capability spend, or provider dispatch. Ollama accepts only `low`, `medium`, and `high`; Claude, Grok, and OpenCode also accept `xhigh` and `max`.
- Disposable-state cleanup settles independently from the authoritative provider outcome. A validated review survives cleanup failure with only the bounded `temporary_state_cleanup_failed` operational warning. Provider and preflight failures remain authoritative, and raw cleanup paths, exceptions, and causes are not retained or published. On POSIX, the next provider run performs a bounded stale-run scan. `data status` applies top-level entry, descendant entry, depth, byte, and deadline limits before reporting attributed counts. Incomplete inventory blocks explicit purge. A same-user process that can rewrite private state remains outside the claimed isolation boundary.
- Provider choice never silently changes. A workspace provider lane serializes provider-capable turns. Short mode and optional summary-consent locks atomically issue one capability for each executable configured reviewer. Each capability binds its exact provider/model configuration, prompt/schema hashes, consent revisions, and deadlines; the locks are released before reviewers run concurrently. A mode or consent mutation commits its new revision and then drains capabilities captured from the revoked revision before returning. Only positive spend settlement counts as drained; PID death and elapsed deadlines do not.
- Timeouts terminate the supervised provider containment domain. On POSIX, `runProcess` launches a detached IPC-liveness supervisor and keeps the actual provider plus ordinary descendants in that supervisor's process group. Catchable cancellation and deadlines use TERM→KILL escalation; the supervisor remains the live group leader until cleanup is consumed, so a promptly exiting provider cannot leave a delayed kill aimed at a recycled group ID. Normal provider-leader exit also terminates remaining in-group descendants, including descendants that inherited provider stdout/stderr, before success can be reported. If the hook parent dies non-catchably, kernel IPC closure makes the supervisor immediately SIGKILL its group. A subprocess that deliberately creates a new session/process group can escape this lifecycle boundary, so POSIX supervision is not a sandbox for a malicious provider binary. The Windows native-supervisor contract creates a future provider suspended, assigns it to a non-breakaway `KILL_ON_JOB_CLOSE` Job Object, and resumes only after assignment; helper path, protocol, SHA-256, and PE architecture are verified before launch. Control records are validated from raw printable-ASCII bytes and their terminal state must match the helper's actual exit code and signal. There is no direct-spawn fallback. The v0.5 platform privacy gate runs before this Windows execution path, so helper verification alone cannot enable live provider contact.
- Automatic mode serializes the entire Stop lifecycle per turn and the provider-capable section per workspace. It waits for short authorization, checks the provider circuit while serialized, and writes a durable attempt marker before capability issuance. Queuing behind another turn cannot burn the attempt fence. The capability moves durably from `issued` to `consumed` before executor entry and is never spendable twice. A retry checks an existing marker before recapturing mutable Stop inputs: if the original receipt exists it is recovered under its original key; otherwise Buddy abstains instead of making a second provider call.
- Three consecutive failures open a 30-minute workspace/provider+model circuit. Circuits are checked and recorded independently. An open or failed lane is never replaced with an unconfigured provider. One successful lane can produce an attributed partial result; zero successful lanes preserve the worker result and produce a degraded terminal state.

## Repository credential hygiene

- Credential values, session material, copied auth entries, raw authorization headers, and provider tokens must never be committed or included in tests, docs, prompt exports, receipts, logs, or release artifacts.
- The validation workflow checks out complete Git history without persisted checkout credentials and runs the commit-pinned Gitleaks action with Gitleaks 8.30.1. The release workflow uses the same pinned scanner setup, disables comments, summaries, and SARIF uploads, then runs a separate directory scan against the exact built public artifact. The local `npm run security:secrets` command scans complete reachable history and the current directory, including untracked candidate files.
- Release construction is manual-only from the protected public source head. It produces a byte-verified positive artifact and a separate parentless artifact-tag Git bundle before any repository write. RC publication creates an attested prerelease and does not claim host acceptance. Final publication additionally requires protected host evidence. Both paths require commit-pinned GitHub provenance attestations for the tarball, distribution bundle, and path-free binding receipt, direct attestation verification, exact source and tag-object reconciliation, and an explicitly approved job with only `contents: write`. Retry handling refuses any existing tag or GitHub Release unless its object identity, metadata, prerelease state, asset set, and asset bytes match the verified candidate.
- Release publication separately runs `npm run security:publication` from a clean, non-shallow checkout with complete refs. The bounded gate scans raw path signatures in every reachable blob, candidate text content, tracked and historical pathnames, commit messages and identities, annotated-tag messages and tagger identities, supplemental commit and tag headers, and reference names. It rejects personal absolute paths, non-public email addresses, Buddy private-state or receipt shapes, invalid UTF-8 metadata, symlinks, Git links, and incomplete or over-limit coverage. GitHub noreply identities and `.invalid` fixture addresses are accepted; another reviewed public contributor address must be explicitly repeated with `--allow-email`. Clean CI additionally runs exact-index `--tree-only` mode. The full-history clean-worktree requirement is not added to ordinary local dirty-tree validation.
- Provider adapter tests construct credential-shaped synthetic fixtures from separated source fragments. This keeps detection coverage without publishing strings that resemble usable credentials or creating noisy public secret-scanning alerts.
- A clean automated scan is a release gate, not a proof that every possible secret format is detectable. The positive release artifact is scanned separately after deterministic construction. Clean source and artifact scans are necessary but not sufficient for publication. The public source branch begins at one reviewed parentless commit with GitHub noreply author metadata and no reachable private-development ancestors. The former private history has a private local backup and is not referenced by public branches or release tags. The in-place ref replacement does not claim that GitHub immediately purged every unreachable or cached private object.
- The public builder resolves a real Git `HEAD`, requires an explicit source commit to equal it, and checks every copied or transformed public input for staged, modified, untracked, or ignored state both before and after construction. Development boundary tests use a separate temporary committed snapshot and never present that synthetic commit as a publishable repository revision.

## Output, persistence, and presentation

- Reviewer output is validated against a closed schema.
- Findings and optional comments must be fully contained in an allowlisted transmitted grounding range. Deletion-only and metadata-only changes in surviving files use explicit synthetic anchors. A complete, bounded, privacy-safe deleted file exposes explicit old-side ranges and baseline line counts; result schema v2 requires `line_side: old` for those citations.
- The summary-claim advisory is validated independently against exact zero-based, end-exclusive UTF-16 offsets into the transmitted summary. Its closed fields cannot express code severity, impact, path, line, evidence, or a technical recommendation, and therefore cannot be promoted into a technical finding. Invalid advisory output becomes advisory abstention without invalidating an otherwise valid technical envelope. Both channels share one inference, so Buddy does not claim the technical model output is invariant to the optional packet.
- Terminal controls, Unicode line/paragraph separators, and bidi overrides/isolates are rejected where structural. Every model-controlled multiline terminal field is continuation-prefixed so it cannot forge a finding header.
- Manual receipt storage is disabled by default. Explicit `--store` retains bounded mode-`0600` metadata and the validated result; `--retain-evidence` additionally requires `--store`. Automatic content-bearing receipts are atomically published for crash-safe continuation replay, then become eligible for deletion after the bounded post-observation or unobserved 24-hour recovery window. A content-free tombstone preserves the review key, terminal state, timestamps, and digests so content expiry can never authorize another provider call.
- Buddy v0.5 writes every new v2 outbox event with `worker_summary: null`. A retained legacy v1 event may contain a summary, but renderer output excludes it by default and exposes it only after explicit local `--include-worker-summary` consent. Completed events can contain hashed session/turn identifiers, bounded display text, validated aggregate results, ordered attributed reviewer outcomes, separately validated advisory/presentation fields, and allowlisted paths. Failed lane details use bounded safe failure codes and never provider stderr. v2 sequences are workspace-monotonic but excluded from the deterministic event ID. A non-extendable 24-hour age threshold applies to v1 and v2 content even when no renderer is registered; acknowledgments can shorten retention.
- `buddy-review.mjs data status` reports bounded workspace-local review and settings counts, provider-attributed temporary counts and bytes, and aggregate shared setup/pet state. `data purge` removes exact hashed workspace review content after confirming review mode is disabled and no live egress capability exists. On POSIX it also removes only attributable non-live provider temporary runs with a complete ownership inventory. Live runs, other workspaces, legacy unattributed runs, provider CLI auth, unresolved setup rollback evidence, and the shared pet install registry/backups remain preserved. `--include-settings` applies only to workspace mode, reviewer/consent, presentation, circuit, and drained egress state.
- Hook errors expose a short diagnostic hash rather than raw stderr or absolute secret paths. CLI diagnostics are terminal-escaped, forced onto one line, and capped at 2,000 characters.
- `buddy-review.mjs doctor --json` is local diagnostic output, not a share-safe report. It intentionally includes absolute workspace and state/registry paths plus the configured reviewer provider/model inventory. Review and redact it before copying any portion into an issue, chat, external model prompt, or shared log.
- Provider prompts and Stop continuations use per-delivery unpredictable closed boundaries around bounded JSON, escape Unicode line separators, instruct the receiver to treat the payload as untrusted quoted data, and forbid continuation edits/tool calls.
- Presentation personality/mood/utterances are closed deterministic local data. Completion XP uses write-once review-key credits and cannot take finding count, severity, or model text as input.

## Known limitations

- Endpoint, fragment, and high-confidence token matching are not activity-history or semantic DLP. They cannot recover a denied source created and removed entirely between captures, decode arbitrary encodings or archives, identify every independently authored credential in allowed source, or exhaustively scan ignored dependency and generated directories such as `build`, `dist`, `vendor`, or `node_modules`. Short or low-entropy secrets, unknown generic formats, nested encodings, YAML block scalars, XML, and uncommon assignment syntaxes are not exhaustively classified.
- A turn-window delta cannot prove exclusive worker authorship if another actor edits concurrently.
- The synchronous Stop review can keep a task active for up to the configured provider deadline; this is an intentional delivery/latency tradeoff for the native-pet MVP.
- A crash after baseline capture but before Stop can leave private ephemeral objects until opportunistic TTL pruning runs on a later eligible capture. The pruner terminalizes stale state before cleanup and never re-authorizes provider egress.
- Circuit, mode, and turn serialization use unique-claim local filesystem locks, not distributed locking. PID reuse can conservatively delay stale-claim recovery rather than admit overlapping owners.
- Renderer delivery is at-least-once within the event retention window. Offline consumers cannot extend the content age threshold. Aged v1 events are compacted under the workspace lock by publishing the reduced migration index before immutable event deletion; producer high-water state prevents sequence reuse.
- Pet package publication and ownership-registry persistence remain separate durable steps, bridged by immutable journals and explicit exact-state reconciliation. Ambiguity is preserved as `needs_attention`. Private JSON contents and parent directories are synced on POSIX, but cross-platform filesystem semantics still do not justify a universal power-loss durability claim.
- Pet ownership is scoped by the canonical physical Codex home. Real-path and symlink-alias invocations share one registry and lock domain; distinct Codex homes cannot overwrite each other's ownership records.
- Guided setup uses immutable intent and per-subsystem receipts. Apply resumes only exact approved before/after states; rollback validates both subsystems before mutating, restores mode first, and recognizes the pet-update midpoint where the new package has been preserved but the prior backup is not yet restored. Unexpected ownership, hashes, backup identities, or mode revisions fail closed. Expired never-started plans and terminal journals older than 24 hours are removed through bounded, lock-protected quarantine; partial, malformed, or unresolved state is preserved. The journal is process-crash recoverable but inherits cross-platform directory-durability and same-user filesystem race limitations.
- Private state-directory ancestors are trusted local configuration. Catalog/package paths are realpath-bound and reject symlinked package components, but the state layer does not yet use directory-handle-relative operations for every ancestor race. POSIX state uses UID and mode checks. The current Node state layer does not inspect or set an explicit user-only Windows DACL for the default path or a custom `CODEX_BUDDY_DATA_DIR` or `PLUGIN_DATA`. Windows privacy readiness remains blocked on real ACL verification, and the platform privacy gate enforces that limitation before live review state is created.
- Continuation publication records a tokenized claim, then `stdout_written` from the write callback, then `observed` from `stop_hook_active`. Codex supplies no acknowledgment token for external visibility, so a crash between visible stdout and durable observation can still cause a later duplicate replay; no replay is possible unless the host invokes Stop again.
- An unresolved capability left by a process crash is intentionally availability-conservative: later mode or summary-consent mutation can time out rather than infer settlement. A positive supervisor-backed recovery/acknowledgment protocol would be required before safe automated removal.
- The Windows control token provides high-entropy protocol correlation, not a hostile same-user security boundary. It remains visible in the helper command line but is excluded from the separately randomized named-pipe name. The parent does not yet bind the first pipe client to the spawned helper PID. Terminal-record/helper-exit coupling narrows forged-success behavior, but a process already running as the same account is outside this containment threat model.
- Windows helper verification provides at-rest path, hash, and architecture integrity before `spawn`; it does not hold an execution-stable file handle across the verification-to-spawn interval. A same-account writer able to replace the trusted plugin tree remains outside the threat model. Resisting that actor would require a stronger ACL, signed-helper, or native open-and-execute design.
- The checked-in Windows helper manifest is deliberately unavailable until a reviewed binary is built and hash-pinned. CI compiles an ephemeral x64 helper and runs the Windows-only runtime gates; source presence alone is not claimed as Windows runtime proof, and ARM64 remains unavailable until separately built and exercised. Exact final-semver packaging is mechanically rejected unless a verified x64 helper is present in the positive artifact.
- A non-catchable provider adapter crash can leave temporary prompt or selected-auth state. Claude, Grok, and OpenCode use an ownership-marked temporary root. On POSIX, a later provider launch can quarantine exact owned runs older than 24 hours only when the recorded PID is no longer live. Cleanup is opportunistic and PID reuse preserves data conservatively. Legacy v1 markers are compatible with POSIX TTL cleanup but intentionally remain unattributed to workspace purge. The v0.5 gate prevents new Windows provider runs; residue from an older build remains visible and preserved because neither a user-only DACL nor safe later ownership is proven. Direct Windows provider privacy readiness remains blocked pending real ACL implementation and CI evidence. None of these limits permit fallback to another provider.

## Reporting

While the repository remains private, report security concerns privately to the owner through GitHub. Private vulnerability reporting is enabled as part of the public visibility transition. Once enabled, use it rather than a public issue for sensitive reports. Never attach real credentials, proprietary patches, or retained evidence.
