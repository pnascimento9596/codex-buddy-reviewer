# v0.5.0-rc.1 release note

## Decision

Ship the hardening work as an evidence-backed release candidate, not as a stable release. The code and generated public artifact can advance through deterministic validation, independent review, protected CI, and publication without pretending that desktop host observations have happened. Human artifact-bound observations for each pet are intentionally deferred until real users or pull requests make that process useful; they are not a public RC launch gate.

## Architectural delta

The main v0.5 change is the egress authorization boundary. v0.4 held mode and optional summary-consent locks through the full remote inference. v0.5 instead serializes provider-capable work in a dedicated provider lane, issues an exact-bound single-use capability under short configuration locks, releases those locks, then durably consumes the capability before entering the executor. A concurrent mode or summary-consent mutation commits its new revision, captures prior-revision capabilities, and returns only after positive settlement.

Continuous mode adds latency overlap without using the screen. After `UserPromptSubmit` stores a stable private Git baseline, one bounded detached worker confirms repository checkpoints and can launch at most two speculative generations. One or two configured reviewers receive the same exact privacy-filtered technical evidence and run independently. Stop captures the final checkpoint, adopts only an exact matching receipt, and otherwise performs the exact final fallback without duplicating an ambiguous attempt. Visible output is one paragraph of at most three sentences and 700 characters; the complete attributed result stays in the private local receipt.

Windows provider launch is now designed around a native authenticated supervisor and a Job Object with `KILL_ON_JOB_CLOSE`. The provider is created suspended, assigned before resume, and denied a direct-spawn fallback. The checked-in source manifest intentionally marks helpers unavailable; Windows CI builds an ephemeral x64 helper and must pass the real runtime process-tree tests before the branch is considered cross-platform green. ARM64 remains unavailable until it has its own build and runtime evidence.

The distribution boundary is now positive rather than exclusion-based. A generated public artifact contains an exact allowlist of runtime files plus Byte, Mochi, Orbit, Bella, and Lupo. All five provenance records grant public redistribution under the repository license. Release provenance fails closed unless the declared source commit is the current repository `HEAD` and every public input matches that commit before and after construction. The exact built directory receives its own pinned Gitleaks 8.30.1 scan before archiving or upload.

## Deliberate limitations

- The technical review and optional worker-summary advisory share one model inference. Their schemas and local validation are separated; causal independence is not claimed.
- Enabling the worker-summary advisory disables speculative background review for that turn because the screened summary does not exist until Stop.
- Endpoint snapshots cannot recover content created and deleted entirely between captures. The high-confidence scanner and denied-content fingerprints reduce risk but are not universal semantic DLP.
- Continuous mode can spend up to two speculative calls plus one exact final fallback per configured reviewer. Final-only mode remains available for lower egress and subscription usage.
- An issued or consumed capability left unresolved by a crash is not deleted because its PID died or deadline elapsed. A later revocation command may time out until a positive recovery protocol exists.
- The v0.5 renderer remains the native Codex pet plus transcript/outbox presentation. No GUI sidecar or network listener is included.
- Manual review storage is opt-in. Automatic content-bearing recovery state and renderer events have bounded retention, while content-free tombstones preserve at-most-once provider semantics after content expires.
- Source inspection and macOS tests do not prove Windows behavior; the Windows Actions job is the runtime gate.
- Automated validation does not prove Codex command discovery, hook trust, visible one-continuation behavior, or pet animation. Those observations remain honest unrun gaps until adoption makes them useful.

## Promotion gate

Promote to `v0.5.0` only when all of the following are true:

1. The frozen exact source head passes the complete local suite, plugin/skill validators, public-boundary verifier, a fresh sealed whole-repository Codex Deep Security Scan, and independent Grok 4.5 plus Claude Opus 4.8 high reviews.
2. GitHub Actions passes Ubuntu, macOS, real Windows x64 helper/runtime tests, and Node 24 compatibility at that same head.
3. A generated artifact is byte-verified after deterministic archive/re-extraction.
4. The positive final artifact contains a reviewed, hash-pinned `win32-x64` Job Object helper, and the exact packaged bytes pass the Windows runtime tree tests. RC artifacts may remain fail-closed while that binary is unavailable; ARM64 may remain explicitly unsupported.
5. The final version/cachebuster commit is landed on the protected default branch, and a manual final-candidate dispatch passes the same complete matrix before producing the exact artifact used for later stable acceptance.
6. Before stable `v0.5.0` publication, one v2 report per `release-manifest.json.public_pet_ids` entry passes the existing machine and named-observer gates and binds to that exact artifact.
7. A second manual dispatch from the unchanged protected default-branch head uses the same final version with `publish: true`. It reruns the complete matrix, rebuilds and reverifies the artifact, validates the exact host-evidence bundle required by the current workflow, attests the tarball, distribution bundle, and receipt, then publishes only the reconciled parentless artifact-only tag plus its byte-matched GitHub Release. No source-tree version tag is created by hand.

Collecting those five named human reports is intentionally deferred and does not block this public RC. It remains a stable-release gate until the protected workflow is deliberately changed and reviewed.
